# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Expo version

This project is on **Expo SDK 54 / React Native 0.81 / React 19**. Expo's APIs change between major versions — read the exact versioned docs at https://docs.expo.dev/versions/v56.0.0/ before writing native/Expo code (per `AGENTS.md`).

## Commands

```bash
npm install            # install deps
npx expo start         # dev server; press i (iOS sim), a (Android), w (web), or scan QR in Expo Go
npx expo start -c      # same, clearing cache — required after changing .env vars
npm run ios            # native dev build on device/sim (needed for push; Expo Go can't do remote push)
npm run web            # web build (the static export Vercel/Netlify serves)

# Regenerate the bundled court/class/pool data (scrapes live sources):
npm run build:data         # everything below, in order
npm run build:courts       # SF Rec & Parks indoor gyms
npm run build:sanbruno     # San Bruno RAC (Google Sheet)
npm run build:outdoor      # outdoor basketball/volleyball/tennis/pickleball/soccer/baseball + fitness courts (DataSF; greater-SF bounds — Camp Mather excluded)
npm run build:reservations # tennis/pickleball booked% (rec.us)
npm run build:directory    # court facility facts (sfrecpark directories)
npm run build:classes      # full ActiveNet catalog, 33 categories (+ Claude title translation if ANTHROPIC_API_KEY set)
npm run build:pools        # swimming pools + schedules parsed from seasonal PDFs (pdfjs-dist)
```

`build:classes` and `build:pools` need network access; `build:classes` optionally calls the Anthropic API (Claude Haiku) to translate new class titles to zh/es when `ANTHROPIC_API_KEY` is set, caching results in `scripts/classes-i18n-cache.json` (degrades gracefully to English without a key).

There is **no test suite, linter, or typechecker** configured — this is a plain JS (not TS) Expo app. "Verifying" a change means running the app (`npx expo start`).

## Architecture

A single-screen Expo/React Native app (also deployed to web as a static export) for finding places to play sports in SF. Three pieces fit together:

### 1. UI — one big `App.js` + a bottom-tab shell

`index.js` → `Root` wraps `App` in `SafeAreaProvider` + `AuthProvider`, all inside `I18nProvider` (see i18n below). `App.js` (~1900 lines) holds the home map screen, all shared state, and the `CourtDetail` card. `BottomNav` switches five tabs via the `tab` state in `App.js`: `home` (map) · `classes` (`ClassesScreen`) · `pools` (`PoolsScreen` — swimming) · `social` (`SocialScreen`) · `profile`. Most other features render as **modals** (`AuthModal`, `RunModal`, `FriendsModal`, `FeedModal`, `SignalModal`, `SessionModal`, …) toggled from `App.js`. On first launch `App.js` shows `components/Onboarding.js` (a full-screen overlay: swipeable value-prop slides → pick interests → location primer → optional account nudge) — gated on the `recreate.onboarded.v1` AsyncStorage flag; returning users skip it and get the immediate location request as before. A **one-time coach mark** (persisted under `recreate.coach.sportfab.v1`, mirroring the loc-banner pattern) then points at the sport FAB on the map, dismissed by tapping it or opening the sport dial. The location prompt fires in context from the location slide (`onEnableLocation`); "Create account" opens `AuthModal` inline in signup mode (its `initialMode` prop), and finishing onboarding — or a successful signup, detected via the auth-state transition — dismisses the overlay onto the map underneath. `finishOnboarding` persists the interest picks. Interest picks are stored on-device via `lib/interests.js` (`loadLocalInterests`/`saveLocalInterests`, key `recreate.interests.v1`) so recommendations personalize even signed-out; `App.js` computes "effective" interests (the signed-in profile's `favorite_sports`/`favorite_categories` if set, else the local picks) and passes them to `SocialScreen` → `RecommendPane`. Long lists (Classes, Pools, activity feed) share `components/ScrollTopFab.js` (`useScrollTop` hook + FAB): a back-to-top arrow that appears on swipe-up-while-scrolled. The Classes tab also has a scroll-linked collapsing header (title/subtitle/info block fades as one unit, then its height collapses, driven off scroll offset for smoothness).

The map is **Leaflet + OpenStreetMap inside a WebView** (no API key/billing). `components/CourtMap.js` is the native (WebView) implementation; `components/CourtMap.web.js` is the web variant — Metro picks `.web.js` automatically on web, so **keep marker/animation/clustering logic in sync across both files**. Markers are **grid-clustered by screen distance** below `DECLUSTER_ZOOM` (nearby courts collapse into a count bubble — orange if any is open now, grey if none; tap to fit-bounds in), re-clustering on `zoomend` only; past that zoom every court renders individually. Open courts draw as the full sport glyph, **closed/no-drop-in courts recede to a small faded dot** (an open/closed hierarchy). Per-sport glyphs are inline SVG; **pickleball has no emoji, so `components/SportGlyph.js` renders a drawn/rasterized lime perforated ball** (`assets/pickleball.png`) for the sport dial while the map markers draw the same ball as SVG.

### 2. `lib/` — feature stores, each with the same Supabase-or-local pattern

`lib/supabase.js` exports a `supabase` client **or `null`** when `EXPO_PUBLIC_SUPABASE_URL` / `EXPO_PUBLIC_SUPABASE_ANON_KEY` are unset. This null-check is the core architectural seam: **every social/shared feature must degrade gracefully when `supabase` is null** — crowd check-ins fall back to on-device `AsyncStorage`, and account/social features hide entirely. Don't assume the client exists.

Each file in `lib/` owns one feature's data access (e.g. `crowd.js`, `reviews.js`, `runs.js`, `signals.js`, `friends.js`, `feed.js`, `chat.js`, `push.js`, `playerCheckins.js`, `favorites.js`, `blocks.js`, `reports.js`). **Trust & safety (App Store UGC requirement):** `lib/blocks.js` (`getBlockedIds` — in-memory-cached, invalidated on block/unblock) is filtered into every social loader (`signals`, `chat`, `feed`, `friends`) so a blocked user's content disappears app-wide; `lib/reports.js` files content reports. Report/block surface on chat messages (long-press) and reviews (report-only — reviews have no `user_id`); Settings has a Blocked-users manager, and account creation requires agreeing to the EULA/privacy terms. **Posting a review requires sign-in when reviews are shared** (Supabase configured): `CourtDetail` shows a "Sign in to write a review" prompt (routing to the Profile tab via `onNeedSignIn`) and RLS enforces it (`reviews` insert policy is `to authenticated` + `auth.uid()` — migration `016`, folded into `schema/02_reviews.sql`); local-only reviews stay open, and the report flow is unchanged. Settings also links the **Privacy Policy / Terms (EULA) / Support** pages; the App Store privacy nutrition label is reconciled against the policy in `docs/privacy-nutrition-label.md`, and the same collected data types are declared in `app.json` → `ios.privacyManifests` (Location omitted — on-device only). **Accessibility:** every icon-only control (sport/filter FABs, recenter, ✕ close buttons, loc-banner dismiss, bottom-nav tabs, review Report) carries an `accessibilityLabel`/`accessibilityRole` (a11y strings under `a11y.*` in i18n). Realtime features merge incoming rows **incrementally by `id`** (e.g. `mergeCheckIn`) rather than refetching. `lib/auth.js` is the `AuthProvider` context (session + profile). `lib/useCourts.js` is the launch-time data loader. `lib/hours.js` holds the open-now / open-gym schedule logic shared between the map filter, Nearby list, and run/signal time pickers. `lib/recommend.js` (`buildRecommendations`) builds the Social tab's auto-revolving **"Recommended for you"** pane (`components/RecommendPane.js`) from bundled data — open-gym games for the user's `favorite_sports` and rec-center classes with openings in their `favorite_categories` (class-category interests, both edited on the profile). Recommendations are **tiered**: (1) if any interests are set, filter to them; (2) else fall back to the user's **check-in history** (`lib/playerCheckins.js` `loadMyStats` → the sports they've logged, surfacing their most-visited court first — `RecommendPane` fetches this when interests are empty and passes it as `history`); (3) else, with neither interests nor history (signed-out or brand-new), recommend **nearby** (any sport/class, closest first). Tapping a game rec opens that court's `CourtDetail` **on the rec's sport** (the tap threads the sport through `onPickCourt(id, sport)`, switching the map's sport so e.g. a ping-pong rec doesn't open on basketball); tapping a class rec — or any card on the Classes tab — opens `components/ClassDetail.js`, a bottom sheet with the class's schedule / location / cost / ages / availability / description (the catalog blurb, or "No description available") + Directions and a Register-on-ActiveNet link. Free walk-in drop-ins that can't be registered online (ActiveNet shows a "View Registration Info" fee placeholder) are flagged `noOnlineReg`, priced **Free**, and get a "just show up at the start time" notice. ActiveNet **delists** cancelled/ended classes, so a re-scrape drops them automatically; on native, `ClassesScreen` also hides any bundled class missing from a healthy live-catalog fetch (`lib/classesLive.js`) so a between-scrape cancellation disappears in real time. **ActiveNet scrape gotchas** (`scripts/build-classes.js`): query **one category id per request** — the multi-id search repeats some categories' items across pages and silently omits others (`lib/classesLive.js` `GROUPS` must stay in sync with the build's id list, or the delist-hiding wrongly hides whole categories on native); drop-in series arrive as one activity **per date** and are collapsed into a single card keyed by the **latest** instance's id (earlier instances delist while the series still runs); drop-ins for map-tracked sports are skipped (`MAP_SPORT_DROPIN_RE`) since their hours are the map's open-gym blocks; "View Fee Details" placeholder fees get real prices from `rest/activity/detail/estimateprice/<id>`. `ClassDetail`'s sheet pins Directions/Register **outside** its ScrollView and caps the ScrollView with a **numeric** maxHeight — native Yoga does not shrink flex children to a parent's percent maxHeight (web does), so a percent-capped sheet alone clips long content unscrollably. The same recommendations feed **interest-based local notifications** (`lib/localNotify.js`, called from `App.js` on launch + foreground): on-device reminders ~30 min before today's matching games/classes, deduped per day in `AsyncStorage`, gated on having interests + notification permission. Server push (while the app is closed) is handled entirely in Postgres: `supabase/schema/07_push.sql` triggers call Expo's push API via `pg_net` (`send_push()`) when a friend posts a signal, plans a run, logs a check-in, or reports a court's crowd level — plus direct pings (someone joined your run/signal, a session got confirmed, a friend request was accepted). The four **broadcast** notifications (check-in, crowd vote, signal, run) are gated on a per-row `notify` flag the client sets from the user's `profiles.share_activity` setting (Settings → "Share activity with friends", default on); when that setting is off, the client prompts per-action (`lib/activityShare.js` `resolveNotify`) so a private user can still share one thing. `send_push()` and `accepted_friend_ids()` are `SECURITY DEFINER` and **revoked from anon/authenticated** (only the trigger functions, running as owner, may call them) — otherwise a client could push spam or enumerate friend lists. `send_push()` sends **one request per token** (migration `013`): Expo 400s a whole request that mixes tokens from different projects/experience IDs, so batching let one stale token silence a broadcast — a real gotcha after re-creating the EAS project left a device with both an old-`hoopmap` and new-`recreate` token. Crowd-update pushes are **rate-limited to one per voter+court per 10 min** via `crowd_notify_log` (migration `014`) — rapid level switches (each an insert + delete) would otherwise push to friends on every tap. `lib/favorites.js` (`useFavorites` hook) stores favorites on-device via `AsyncStorage` (works fully signed-out, like crowd check-ins) as a **court id → sport** map: a court is favorited *for the sport you were viewing* when you starred it. The sport speed-dial's ⭐ **Favorites** entry switches the map into a personal view of just those courts, each marker/card judged open **for its favorited sport**; the card's star (`toggle(id, sport)`) saves/removes the court for the displayed sport. `lib/sports.js` defines the tracked **playable** sports (`SPORTS`: basketball, volleyball, ping pong, badminton, pickleball, tennis, **soccer**, **baseball** — soccer & baseball are outdoor-only field sports. SF fields are strictly designated by DataSF type: soccer = `Soccer Field` + `Multi-Use Turf`, baseball = `Ball Field` (diamonds). A single field is never both; a park with a separate pitch & diamond unions both onto one pin, exactly like a park with both a basketball & tennis court) plus the `ANY_SPORT` (`'any'`) sentinel — "just down for rec", a sport-less **"down to play"** signal (no real schedules key off it, so the session sheet falls back to a default sport for court/time suggestions). It also exports `WEIGHT_ROOM` (`'weightroom'`), `GOLF` (`'golf'`), `MAP_SPORTS` (`SPORTS` + both, for the map dial), and `PLAN_SPORTS` (`SPORTS` + both, for the social composers): weight room and golf are **facility views**, not sports — the weight room is scraped into `dropins.weightroom` like a sport, and the 6 SFRPD golf courses are hand-curated in `data/manual-courts.js` with daylight-hours `dropins.golf` weeks plus a per-course `golf` block (holes/par/yardage, a short course description, curated green fees like the pool `FEES` tables — bump ~annually — and tee-time booking + official-website URLs) that `CourtDetail` renders; golf-only filter chips (9/18 holes, beginner, driving range) ride the generic `AMENITIES` machinery, self-hiding outside the golf view because only golf entries carry `c.golf`. Both views are selectable in the map's sport speed-dial (`MAP_SPORTS`), so all the generic marker/open-now/card machinery works. Both are kept out of `SPORTS` — so **favorites/profile** never treat them as sports, and `isPlayableSport(id)` guards the map→`SocialScreen` hand-off (`App.js` coerces a facility-view map sport back to `DEFAULT_SPORT`) — **but both social composers can plan/signal a lifting session or a round** via `PLAN_SPORTS` (Plan a session in `RunModal`, "I'm down" in `SignalModal`). Only Mission, Minnie & Lovie Ward, and Glen Park currently publish weight-room drop-in hours; the weight-room view also includes the 12 DataSF **outdoor fitness courts** (`Adult Fitness Court/Course`, mapped to `dropins.weightroom` by `build-outdoor-courts.js`). Both planned runs and signals carry a `sport`; a run's/signal's sport scopes its court/time suggestions, and signals may be `ANY_SPORT`. **Plan a session (`RunModal`) is for any planned session, not just casual pickup** — it offers an Indoor/Outdoor/Either filter over the court list for sports that have both (the map's `PLACE_OPTS`), and requires a concrete court + time. **A "down to play" signal (`SignalModal`) can optionally seed location up front**: an indoor/outdoor `place` and/or a preferred court (`rec_signals.place` + `pref_court_id`, both nullable — blank = "let friends decide", added in migration `015`), surfaced on the session sheet as a "Prefers: …" line. When responding to a signal, a participant suggests an **activity → place → time** (`rec_signal_participants.proposed_sport` + `proposed_court_id` + `proposed_at`); the host confirming a suggestion promotes its sport onto the signal (so an `ANY_SPORT` signal becomes concrete). Joining a run or signal makes you a member of its `chat_messages` group thread (`lib/chat.js`) — `chat_messages` holds run/signal group chats and 1:1 direct chats in one RLS-scoped table; joining a signal from the session sheet also posts an announcement into that chat.

**i18n (`lib/i18n.js`):** the app is fully localized to English / 中文 / Español. `I18nProvider` holds the chosen language (persisted in `AsyncStorage` under `recreate.lang`); React components read `const { t, lang } = useI18n()`. Plain non-React modules (`datetime`, `crowd`, `hours`, the `lib/*` error messages) translate through the module-level `tg(key, vars)` helper, which mirrors the current language into a global so it works without context. All strings live in one `STRINGS` dict keyed by language — **keep en/zh/es at full key parity** (a quick `eval`-based count check catches drift). Composed sentences use `{placeholder}` interpolation, not string concatenation. External data (court/pool names, addresses, scraped class/schedule text) stays in its source language; scraped **class titles** are the exception — they're pre-translated at build time (see `build:classes`).

Supabase DDL lives in `supabase/schema/` (canonical, numbered 01→10 — `09_account_deletion.sql` is the SECURITY DEFINER `delete_account()` RPC behind Settings → Delete account; `10_moderation.sql` is trust & safety: `blocked_users` + `content_reports` — **run in order** on a fresh DB, later domains depend on earlier) and `supabase/migrations/` (deltas for existing DBs — the latest is `016_reviews_require_auth.sql`, which tightens the `reviews` insert policy to require an authenticated user). When adding a feature with a new table, add a numbered schema file and document its dependency order in `supabase/README.md`.

### 3. `data/` + `scripts/` — a generated, self-refreshing data pipeline

**`data/courts.js`, `data/courts.json`, `data/outdoor-courts.js`, `data/classes.js`, `data/pools.js`, etc. are GENERATED — never edit them by hand.** Each is produced by the matching `scripts/build-*.js`, which scrapes a live source (sfrecpark.org via cheerio, DataSF `ib5c-xgwu`, rec.us API, a Google Sheet, ActiveNet). The two hand-authored data files are `data/manual-courts.js` (fully-static courts merged in at runtime by `lib/useCourts.js`, deduped by `id` — this is where the 6 golf courses live) and the `CENTERS` table inside `scripts/build-indoor-courts.js`.

Build resilience pattern (shared by every script): each source falls back **live → cache → curated**, with a `scripts/*-cache.json` last-good snapshot and a validation gate that aborts the build (keeping old data) if too few records scrape — so an upstream site redesign fails loudly instead of publishing empty data.

**Pools are the unusual one (`scripts/build-pools.js` → `data/pools.js`):** the 9 SF public pools post their weekly swim schedules only as *seasonal PDFs* (re-issued every few months) on their facility pages. The script discovers each pool's current schedule PDF link live, downloads it, extracts positioned text via `pdfjs-dist` (a build-only devDep — not bundled into the app), and reconstructs the weekly grid geometrically (merge text fragments → map cells to day columns by x → pair each activity label with the time below it → classify into a session `kind`). Output is `sessions[dow] = [{ kind, start, end }]` (0=Sun..6=Sat, minutes-from-midnight, same convention as court schedules), plus the city-wide `POOL_FEES` and `POOL_CLOSURES`. A facility posting separate warm-/cool-pool PDFs (North Beach) gets each session tagged `pool: "warm" | "cool"` by source PDF — the tag is part of the cross-PDF dedupe key (both pools can run the same activity at the same time) and `PoolsScreen` renders the groups under ♨️/❄️ headers. Pool coordinates/addresses/phones and the `season`/closure labels are **curated** in the script's `META`/`FEES`/`CLOSURES` tables (pages have no lat/lng; fees change ~annually, not seasonally) — bump these by hand each season. `PoolsScreen` renders a localized label per `kind` and links out to the official PDF as the source of truth.

`lib/useCourts.js` loads data **bundled (instant) → cached → remote** (`EXPO_PUBLIC_COURTS_URL`) so the app renders offline immediately then revalidates. It also merges the auxiliary generated maps onto each court: `reserved` (booked%), `directory` (facility facts).

GitHub Actions crons re-run the builds and commit only when the generated data changed: `refresh-schedules.yml` runs `build:data` weekly (courts, reservations, directory, classes, **pools**), `refresh-classes.yml` re-scrapes classes every 6h, and `refresh-reservations.yml` re-scrapes rec.us occupancy every 3h (bookings change hourly, so the weekly snapshot goes stale fast). On native the court card also fetches a **live** rec.us reading for the open location (`lib/reservationsLive.js`, mirrors `lib/classesLive.js`) and overlays it onto the snapshot's `reserved[sport]` — so "% booked at 3 PM" is real-time; web CORS-falls back to the snapshot and the card labels freshness ("Live from rec.us" vs "as of M/D"), only asserting a hard "🔴 Fully booked" when the reading is live or the snapshot is <6h old. **When adding a new generated file, also add it to the workflow's `FILES` commit list** or the refresh will run but never commit it. Because reservation/class/pool slots are date- or season-keyed and go stale, this refresh is what keeps "right now" accurate.

## Environment & config

All runtime config is `EXPO_PUBLIC_*` (inlined at build time, client-safe — the Supabase anon key is protected by RLS). Copy `.env.example` → `.env`. The app runs fully **signed-out with no env vars set** (bundled data + local check-ins); env vars progressively enable remote data and social features. Restart with `npx expo start -c` after changing `.env`.

`ANTHROPIC_API_KEY` is **not** an app var — it's used only by `build:classes` (CI/local) to translate class titles. Never expose it to the web build; only the `EXPO_PUBLIC_*` vars belong in the host's env.

## Deploy (web)

**Vercel Analytics** is mounted web-only via `components/WebAnalytics.web.js` (renders `@vercel/analytics/react`), with a native no-op `WebAnalytics.js` — same `.web.js` platform-split as `CourtMap`, so the DOM-based lib never enters the native bundle. Use the `/react` entry, not `/next` (this is a react-native-web SPA). The launch splash is `expo-splash-screen` (config in `app.json` plugins → `assets/splash-icon.png`, the running stick figure). The app is **iPhone-only** (`ios.supportsTablet: false`) — runs on iPad in compatibility mode, no iPad assets needed.

The web target is a **static SPA export** (`npx expo export --platform web` → `dist/`), served by Vercel (`vercel.json`) or Netlify (`netlify.toml`) — both mirror the same build command, `dist` output, and SPA rewrite of all paths to `index.html`. Set the three `EXPO_PUBLIC_*` vars in the host dashboard (a CI build has no local `.env`). Don't run both hosts as primary auto-deploys at once. Note: the live ActiveNet class-availability fetch (`lib/classesLive.js`) is blocked by CORS on web and falls back to the bundled baseline — expected; native isn't bound by CORS.

## Deploy (native) — crash reporting & OTA updates

**Sentry** (`@sentry/react-native`) is native-only via `lib/crash.js` + a no-op `lib/crash.web.js` (same platform split as `WebAnalytics`), gated on `EXPO_PUBLIC_SENTRY_DSN` — unset means fully disabled, so dev and web are unaffected. `index.js` registers `withCrashReporting(Root)`; `metro.config.js` wraps Expo's default config with `getSentryExpoConfig` (harmless when disabled). Init is privacy-minimal: `tracesSampleRate: 0`, `sendDefaultPii: false`, and **never call `Sentry.setUser`** — crash data is declared "not linked to identity" in `app.json` `privacyManifests` + `docs/privacy-nutrition-label.md` + the privacy policy, and linking it would falsify all three. Sourcemap upload needs the account-specific bits: replace the bare `"@sentry/react-native"` plugin entry in `app.json` with `["@sentry/react-native/expo", { "url": "https://sentry.io/", "organization": "<org>", "project": "<project>" }]` and set `SENTRY_AUTH_TOKEN` as an EAS secret.

**EAS Update (OTA)**: `app.json` sets `updates.url` + `runtimeVersion.policy: "fingerprint"`, and the `preview`/`production` build profiles in `eas.json` pin matching channels. Ship a JS-only fix with `npm run update:prod` (= `eas update --channel production`) — installed apps fetch it on next launch, no App Store review. The fingerprint policy means an update only reaches builds with identical native code, so **adding/upgrading any native module requires a new store build** (`eas build`) — the OTA update will silently not apply to older binaries. Both features touch native config, so the first build after enabling either must be a full rebuild (`npm run ios` locally or `eas build`).
